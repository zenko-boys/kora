# Security Findings

Generated by security review on 2026-06-02.

---

## HIGH

### [ ] JWT Audience Validation Disabled
**File:** `src/Kora.Api/Infrastructure/Auth/AuthExtensions.cs`

`ValidateAudience` is silently set to `false` when `Auth:Audience` is empty â€” which is the current config in all environments. Any Clerk-signed JWT for any app on the same Clerk instance is accepted as fully valid.

**Fix:** Set `Auth:Audience` to the Kora Clerk application's client ID via Fly secrets (`fly secrets set Auth__Audience=... --app koraplay-api`). Remove the `IsNullOrWhiteSpace` bypass â€” throw if the value is missing rather than disabling the check.

---

### [ ] Live Resend API Key on Disk
**File:** `src/Kora.Api/appsettings.Local.json`

A live Resend API key is stored in `appsettings.Local.json`. If this file is ever accidentally committed, included in a build artifact, or the machine is compromised, an attacker can send emails from the domain to any address.

**Fix:** Rotate the key immediately. Switch to `dotnet user-secrets` for local development so secrets never live in the project directory tree.

## MEDIUM

### [ ] Wildcard CORS Policy
**File:** `src/Kora.Api/Infrastructure/Cors/CorsExtensions.cs`

`AllowAnyOrigin().AllowAnyHeader().AllowAnyMethod()` is applied unconditionally. A dead `IsLocalhost` helper exists in the same file. Any origin can make cross-origin requests to the API.

**Fix:** Replace `AllowAnyOrigin()` with `WithOrigins("https://app.koraplay.com", "http://localhost:3000")`.

---

### [x] Internal Exception Messages Exposed in Responses
**File:** `src/Kora.Api/Common/Errors/GlobalExceptionHandler.cs`

`InvalidOperationException.Message` and `UnauthorizedAccessException.Message` are returned verbatim as the `Detail` field of HTTP responses. Any future exception thrown by third-party code (EF Core, Npgsql, etc.) could expose internal infrastructure details.

**Fix:** Introduce a custom `DomainException` base class. Only expose `.Message` from known domain exceptions; return a generic "Bad Request" detail for all others and log the real message server-side.

---

### [x] Privilege Escalation via Email Claim
**File:** `src/Kora.Api/Infrastructure/Auth/UserContext.cs`

Admin role is re-evaluated on every request by comparing the JWT's `email` claim against `Auth:AdminEmails`. Anyone who registers a Clerk account with a matching email is automatically promoted to Admin on their first request â€” no additional verification step.

**Fix:** Manage admin role assignments in the database via a protected endpoint. If the config-driven approach is kept, only evaluate at JIT-creation time, not on every request.

---

### [ ] Booking Existence Enumerable Before Authorization Check
**File:** `src/Kora.Api/Features/Bookings/DeleteBooking/DeleteBookingHandler.cs` (same pattern in `JoinBookingHandler`, `LeaveBookingHandler`)

The handler fetches the booking before checking authorization. An authenticated member receives distinct error responses depending on whether a given GUID exists â€” leaking the existence of arbitrary bookings.

**Fix:** Return a generic 404 for both "not found" and "unauthorized" cases when the user has no relationship to the booking's club.

---

### [x] Unencoded User Input in HTML Email Templates
**File:** `src/Kora.Api/Infrastructure/Email/EmailTemplate.cs`

Email bodies were built via raw C# string interpolation with no HTML encoding, making it easy to accidentally inject user-supplied strings (e.g. `booking.Description`) unencoded into email HTML.

**Fix applied:** `EmailTemplate.Wrap` now HTML-encodes `title`. A public `EmailTemplate.Encode(value)` helper was added â€” use it whenever interpolating a user-supplied string into an email body.
